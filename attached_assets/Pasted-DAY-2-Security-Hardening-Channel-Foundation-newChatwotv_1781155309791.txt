Task 2.1 — Fix WhatsApp Webhook Verification

الهدف

إصلاح WhatsApp webhook verification بشكل آمن.

المطلوب

* لا تستخدم return true.
* اقرأ raw body قبل JSON parsing.
* تحقق من header:

X-Hub-Signature-256

* استخدم HMAC-SHA256:

sha256={hmac(rawBody, appSecret)}

* appSecret يجب أن يأتي من channel config أو env بطريقة آمنة.
* أضف GET handler للتحقق من Meta webhook challenge:

hub.mode
hub.verify_token
hub.challenge

* verify_token يجب أن يطابق channel.config.verifyToken.
* لو verification فشل: return 403.
* لو channel غير موجود: return 404 أو 400 بشكل واضح.

الملفات المتوقعة

src/server/channels/providers/stubs.ts
src/server/channels/providers/whatsapp.ts
src/app/api/channels/whatsapp/webhook/route.ts
src/server/channels/incomingPipeline.ts

لو adapter موجود في ملف مختلف، استخدم البنية الحالية ولا تكسرها.

Task 2.2 — Fix Facebook Webhook Verification

الهدف

تأمين Facebook Messenger webhook.

المطلوب

* لا تستخدم return true.
* نفذ HMAC-SHA256 باستخدام:

X-Hub-Signature-256

* أضف GET handler لـ Meta webhook challenge.
* تحقق من verify_token.
* اربط webhook بصفحة Facebook الصحيحة من خلال:

pageId
channel.config.pageId

* فشل التحقق = 403.
* channel غير موجود = 404 أو 400.

الملفات المتوقعة

src/server/channels/providers/facebook.ts
src/app/api/channels/facebook/webhook/route.ts
src/server/channels/incomingPipeline.ts

Task 2.3 — Fix Super Admin Security

المشكلة

حاليًا requirePlatformAdmin() يعتمد على role === "admin"، وهذا يسمح لأي tenant admin بالدخول إلى /admin/*.

المطلوب

* أضف field جديد في User model:

isSuperAdmin: boolean

* default = false.
* عدّل requirePlatformAdmin() ليعتمد فقط على:

session.user.isSuperAdmin === true

أو يجلب المستخدم من DB ويتحقق من isSuperAdmin.

* لا تعتمد على tenant role.
* حدّث NextAuth session/JWT ليشمل isSuperAdmin إن احتاج.
* أضف script آمن لتعيين أول super admin من env:

SUPER_ADMIN_EMAIL

مثال:

npm run make:super-admin -- email@example.com

أو script مباشر داخل scripts/.

الملفات المتوقعة

src/lib/models/user.ts
src/lib/auth.ts
src/lib/authz.ts
src/app/admin/
src/app/api/admin/
scripts/make-super-admin.ts

Task 2.4 — Implement RBAC Permission Enforcement

المشكلة

Permission system موجود لكنه غير مطبق فعليًا على API routes.

المطلوب

أنشئ helper:

requirePermission(session, permission)

أو:

assertPermission(user, permission)

ويستخدم permission map الموجود في:

src/server/permissions/

طبّق الصلاحيات على الأقل على هذه routes

/api/inbox/conversations/[id]/reply       -> inbox.reply
/api/inbox/conversations/[id]/assignment  -> inbox.assign
/api/knowledge/*                          -> knowledge.manage
/api/settings/*                           -> settings.manage
/api/channels/*                           -> settings.manage أو channels.manage لو موجودة
/api/billing/*                            -> billing.manage أو billing.read حسب العملية

قواعد مهمة

* viewer لا يستطيع reply.
* agent يستطيع reply لكن لا يدير billing.
* manager يستطيع assign.
* owner/admin يستطيعان إدارة settings.
* super admin لا يعني tenant permission تلقائيًا إلا في admin context.

المطلوب توثيقه

أضف جدول صلاحيات في:

docs/architecture/RBAC_ENFORCEMENT.md

المشكلة

الملف المتوقع

Task 2.6 — Implement WhatsApp sendMessage

المشكلة

WhatsApp sendMessage() يرجع success بدون إرسال فعلي.

المطلوب

افصل WhatsApp adapter في ملف مستقل:

src/server/channels/providers/whatsapp.ts

ونفذ:

sendMessage(params)

باستخدام Meta Cloud API:

POST https://graph.facebook.com/v18.0/{phone_number_id}/messages
Authorization: Bearer {access_token}

دعم MVP

* text messages.
* template-ready structure.
* parse external message id.
* handle errors.
* update message delivery status.
* لا تخزن أو تطبع access token في logs.
* access token يكون encrypted at rest، ويفك server-side فقط.

حالات الخطأ

تعامل مع:

* invalid token.
* phone_number_id missing.
* 24-hour window violation.
* rate limit.
* Meta API error response.
